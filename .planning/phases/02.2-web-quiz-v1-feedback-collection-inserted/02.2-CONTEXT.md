# Phase 2.2: Web Quiz v1 & Feedback Collection - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

A deliberately plain public web quiz (Vite + React + React Router on Cloudflare Pages) that lets anyone play a pub quiz against the current 309-question library and captures per-question play data plus feedback to seed calibration. Dress rehearsal for Phases 3 + 7 ‚Äî the Supabase anon auth flow, question fetch, quiz state machine, answer reveal, and scoring all transfer verbatim to the native app.

Scope boundary (from roadmap, do not revisit): Vite + React + React Router, Cloudflare Pages, Supabase client in-browser, anonymous auth, minimal styling, system fonts, no accounts/profiles/leaderboards/polish/gamification.

</domain>

<decisions>
## Implementation Decisions

### Session identity & auth
- **D-01:** Use Supabase anonymous auth (`signInAnonymously()`) on first load. JWT persisted in localStorage via the Supabase client's default storage. This is the session identity for RLS.
  - Rationale: real tamper-resistant JWT, Phase 3 lifts this 1:1, enables `auth.uid() = session_id` RLS.
- **D-02:** `session_id` column on both new tables = `auth.uid()` of the anonymous user. No parallel localStorage UUID.

### RLS model for new tables
- **D-03:** `question_plays` and `quiz_sessions` are **insert-only from the browser**. No SELECT policy for the anon role on either table.
  - Insert policy: `auth.uid() IS NOT NULL AND session_id = auth.uid()`.
  - Score summary on the end screen is computed client-side from in-memory state accumulated during the quiz ‚Äî never re-read from the DB.
  - Rationale: smallest possible data surface, no leak vector, score is already known on the client by the time the quiz ends.

### Write timing & failure handling
- **D-04:** Insert each `question_plays` row **live**, at the moment the user taps a feedback button (good / bad / confusing / next) on the reveal.
  - Rationale: abandoned/ragequit quizzes still yield rows, which is exactly the signal we want. Batching at end-of-quiz would drop "left on Q7" data entirely.
- **D-05:** Insert the `quiz_sessions` row once, at the end screen, after the user has seen the score and submitted (or skipped) the overall rating + free-text feedback.
- **D-06:** On Supabase insert failure (offline, network, RLS error), push the row to a **localStorage outbox** (`question_plays_outbox`, `quiz_sessions_outbox`). On the next successful insert ‚Äî or on next page load ‚Äî flush the queue. Never blocks the UI, never drops data.
  - Outbox entries are plain JSON. Cap size at something sane (e.g. 500 rows) and drop oldest on overflow.

### Landing page ‚Äî category picker & options
- **D-07:** Category picker shows the **12 seed themes from Phase 1 plus a "General" option**. General = any published question across all categories.
- **D-08:** Difficulty = Easy / Medium / Hard, mapping to the Calibrator's current tier labels (hardcoded in a config module for v1 ‚Äî swap to % bands when backlog 999.8 lands).
- **D-09:** Number of questions = 5 / 10 / 15 / 20.
- **D-10:** Question selection within a tier for v1 is **random sample** (no session-awareness, no recent-play filtering). Roadmap lean ‚Äî no discussion needed.

### End-of-quiz rating
- **D-11:** Overall rating is **three faces with text labels** (üòÅ Good / üòê Okay / üòû Bad) ‚Äî icons accompany text, never replace them (accessibility rule from the roadmap, same as the per-question feedback buttons).
  - Stored on `quiz_sessions.overall_rating` as a text enum: `'good' | 'okay' | 'bad'`. Not the integer `1-5` the roadmap originally sketched.
- **D-12:** Single optional free-text box below the faces ‚Äî "Anything to tell us?" ‚Äî no placeholder, no prompts. Stored as `quiz_sessions.feedback_text` (nullable).
- **D-13:** Order on end screen: score summary first, rating + text second (people are honest once they've seen how they did).

### Share behaviour
- **D-14:** Share button copies a URL with **pre-filled setup query params**, e.g. `/?cat=film&diff=hard&n=10`. Friend lands on the setup screen with those values already selected but can change them before hitting "Play".
  - Rationale: "try this exact round" is the actual share intent.
- **D-15:** No result-snapshot URLs (no server-side state, no encoding score). Out of scope for v1.

### Hosting
- **D-16:** **New Cloudflare Pages project**, shipped on the free `*.pages.dev` subdomain for v1 (e.g. `pub-quiz-web.pages.dev`). No custom domain yet ‚Äî move to a real domain later when sharing widens.
- **D-17:** Wrangler config lives in the new project directory. Sibling to the existing pipeline repo, not inside it ‚Äî or as a top-level `apps/web/` (planner decides the monorepo layout).
- **D-18:** Phase 2.3 admin app will be a **separate** Cloudflare Pages project on its own `pages.dev` subdomain, sharing the same Supabase backend but nothing else.

### Privacy
- **D-19:** One-line privacy notice on the landing page: "We log which answers you pick to improve questions. No personal data." No cookie banner, no signup prompt, no modal.

### Claude's Discretion
- Exact file/folder layout inside the Vite app ‚Äî planner picks.
- Whether to use TanStack Query in the web app or just raw `fetch`/`supabase-js` calls. Given the "deliberately plain" ethos, raw calls are probably fine ‚Äî but planner to decide.
- Styling approach ‚Äî plain CSS modules or a minimal utility layer. No NativeWind, no design system (that's Phase 4).
- Quiz state machine shape ‚Äî useReducer, useState, or a small Zustand store. Whatever makes the Phase 7 lift cleanest.
- How the question-fetch query looks exactly (single RPC vs. PostgREST filter) ‚Äî planner + research.
- Outbox flush trigger details (on mount, on next success, on interval) ‚Äî any reasonable mix is fine.
- Exact SQL for the two new migrations ‚Äî planner writes.

</decisions>

<specifics>
## Specific Ideas

- **Time-to-answer accuracy:** roadmap mandates that `time_to_answer_ms` counts only active-tab time ‚Äî `document.visibilityState === 'visible'` AND `window` has focus. Accumulate via `visibilitychange` + `focus`/`blur` listeners. Pause on blur, resume on focus. This is a hard requirement, not a preference.
- **Per-question feedback UX:** the four buttons below the reveal (üëç Good / üëé Bad / ü§î Confusing / Next ‚Üí) both log feedback and advance to the next question. One tap, not two. Icons accompany text labels ‚Äî never replace them.
- **End screen order:** score first, then rating/feedback. People are honest once they've seen how they did.
- **Share intent:** "try this exact round" ‚Äî pre-filled query params, not bare landing URL.
- **Outbox must survive tab-close:** localStorage, not in-memory. Flush on next page load.

</specifics>

## Deferred Ideas

- **Result snapshot URLs** (shareable score pages) ‚Äî not v1. Would need server-side state or signed score tokens.
- **Tier thresholds as % bands** ‚Äî hardcoded to Calibrator tier labels for v1. Swap to observed-score bands when backlog 999.8 lands.
- **Resume-quiz-on-refresh** ‚Äî requires SELECT RLS on own rows, and a session state model we don't need yet. Out of scope.
- **Integer star ratings** ‚Äî replaced with three-face enum. If future analysis needs finer granularity, add a separate scale later.
