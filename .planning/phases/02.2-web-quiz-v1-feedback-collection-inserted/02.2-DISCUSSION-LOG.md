## Session identity & RLS

### Q1: How should anonymous sessions be identified for the web quiz?

| Option | Description | Selected |
|--------|-------------|----------|
| Supabase anon auth | signInAnonymously(), JWT in localStorage, RLS uses auth.uid() = session_id. Real JWT, Phase 3 lifts 1:1. | ✓ |
| Plain localStorage UUID | crypto.randomUUID() on first load. Zero auth overhead but trivially spoofable and Phase 3 throws it away. | |
| Both — UUID now, anon auth later | Ship UUID for v1 speed, migrate in Phase 3. RLS model changes twice. | |

**User's choice:** Supabase anon auth (recommended)
**Notes:** Locks in the Phase 3 reuse — auth flow transfers verbatim.

### Q2: What RLS policy should question_plays / quiz_sessions have?

| Option | Description | Selected |
|--------|-------------|----------|
| Insert-only, no select | Anon role can INSERT rows tagged with own session_id. No SELECT at all. Score computed client-side from in-memory state. | ✓ |
| Insert + self-select | Insert + SELECT only own rows. Enables resume-on-refresh. | |
| Insert + read own session only | Insert anything with own session_id; SELECT limited to own quiz_sessions row. | |

**User's choice:** Insert-only, no select (recommended)
**Notes:** Smallest data surface. Score is already known in memory by the time the quiz ends.

## Write timing & failures

### Q3: When should question_plays rows be written to Supabase?

| Option | Description | Selected |
|--------|-------------|----------|
| Live on each answer | Insert on feedback button tap. Abandoned quizzes still yield data. | ✓ |
| Batch at end of quiz | Collect in memory, bulk-insert on end screen. Loses rage-quit signal. | |
| Live for plays, batch for session | Hybrid — plays live, session row at end. | |

**User's choice:** Live on each answer (recommended)
**Notes:** "Ragequit on Q7" is exactly the signal we want to keep.

### Q4: What should happen if a Supabase insert fails?

| Option | Description | Selected |
|--------|-------------|----------|
| Queue in localStorage, flush on next success | Outbox pattern, never drops data, handles flaky wifi and tab-close. | ✓ |
| Retry 2x then drop silently | Exponential backoff, give up. Loses data on genuine offline. | |
| Drop immediately, log to console | Fire-and-forget. Simplest, loses most data. | |

**User's choice:** localStorage outbox (recommended)
**Notes:** Outbox lives in localStorage so it survives tab close.

## Category picker + rating shape

### Q5: What should the category picker offer?

| Option | Description | Selected |
|--------|-------------|----------|
| 12 seed themes + General | All 12 Phase 1 seed themes + 'General' (any published). | ✓ |

**Notes:** Matches what the pipeline was built around. No curation decision needed.

### Q6: What shape should the end-of-quiz overall rating take?

| Option | Description | Selected |
|--------|-------------|----------|
| Three faces 😁😐😞 | Enum good/okay/bad, lower cognitive load. | ✓ (with text labels) |
| Stars 1–5 | Integer, familiar but noisy. | |
| Both — faces primary, optional stars | Over-engineered for v1. | |

**User's choice:** Three faces WITH text labels
**Notes:** User explicitly added the text-label constraint — consistent with the per-question feedback buttons and the roadmap accessibility rule ("icons accompany text labels, never replace them"). Stored as text enum `'good' | 'okay' | 'bad'` on `quiz_sessions.overall_rating`, replacing the integer `1-5` the roadmap originally sketched.
