# Phase 2.2: Web Quiz v1 & Feedback Collection - Research

**Researched:** 2026-04-07
**Domain:** Vite + React SPA on Cloudflare Pages, Supabase anonymous auth + insert-only RLS
**Confidence:** HIGH on stack and patterns, MEDIUM on a few edge cases (flagged inline)

## Summary

This phase drops the first frontend into the repo: a deliberately plain Vite + React + React Router SPA on Cloudflare Pages that reads published questions from the existing Supabase project, plays a quiz, and writes two new insert-only tables (`question_plays`, `quiz_sessions`) scoped to the anonymous user's `auth.uid()`. Everything the planner needs is locked by CONTEXT.md — stack, auth, tables, styling, outbox, timer. What the planner still picks is layout, exact SQL, state machine shape, and test strategy.

The single biggest schema-reality check: **there is no `question_options` table and no `top_level_category_id` or `difficulty_tier` column in the existing schema.** The roadmap/context references these as if they exist. They do not. Questions carry `correct_answer TEXT`, `distractors JSONB` (array of 3), a single `category_id`, and `difficulty TEXT` ('easy' | 'normal' | 'hard'). Calibration tiers live on `calibration_percent NUMERIC`, not a tier enum. The planner must reconcile this — options are shaped inside the app from `correct_answer + distractors`, and "difficulty tier" maps to the existing `difficulty` enum for v1 (or to `calibration_percent` bands if the planner wants). **This is the most important finding.**

**Primary recommendation:** Drop a `apps/web/` Vite + React + TS project, plain CSS modules, React Router v6 with three routes (`/`, `/play`, `/done`), `useReducer` for the quiz state machine (easy Phase 7 lift), raw `supabase-js` (no TanStack Query), insert-only RLS with `WITH CHECK (auth.uid() IS NOT NULL AND session_id = auth.uid())`, and `.insert()` WITHOUT `.select()` to avoid the RLS-SELECT-after-insert trap.

## Vite + React Router + Cloudflare Pages Setup

### Scaffold
```bash
cd apps && npm create vite@latest web -- --template react-ts
cd web && npm install
npm install react-router-dom @supabase/supabase-js
```

Vite 6 / React 19 / React Router v7 are the current-as-of-2026 defaults. React Router v7 kept the v6 data-router API under a new package name (`react-router`, formerly `react-router-dom`). For a simple SPA both work. **Recommend:** use whichever `npm create vite@latest` installs by default today and pin.

### SPA routing fallback on Cloudflare Pages
Cloudflare Pages serves static files. A `<BrowserRouter>` app needs a catch-all that returns `index.html` for unknown paths, otherwise `/play` returns 404 on refresh.

**Use `public/_redirects`:**
```
/*    /index.html   200
```
That single file, placed in `apps/web/public/`, gets copied to `dist/_redirects` by Vite and Pages honours it. This is the canonical SPA fallback for Pages.

**Alternative:** `_routes.json` if you later add Functions. Not needed for v1.

### Env vars for Supabase URL + anon key
Vite exposes `import.meta.env.VITE_*` at build time. In Cloudflare Pages, set two **build-time environment variables** (Production + Preview):
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

These get baked into the JS bundle, which is fine — the anon key is designed to be public and RLS is the real gatekeeper. **Never** put the service-role key in the browser bundle; this is the single most important security rule and is already implied by the architecture but must be stated in the plan.

Local dev: `.env.local` in `apps/web/` (git-ignored).

## Supabase Anonymous Auth in the Browser

### API shape
```ts
import { createClient } from '@supabase/supabase-js';

export const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY,
  {
    auth: {
      persistSession: true,        // default true; writes to localStorage
      autoRefreshToken: true,      // default true
      detectSessionInUrl: false,   // no OAuth redirects, safe to disable
      storage: window.localStorage,
    },
  }
);

// On first load, before any data query:
export async function ensureSession() {
  const { data: { session } } = await supabase.auth.getSession();
  if (session) return session;
  const { data, error } = await supabase.auth.signInAnonymously();
  if (error) throw error;
  return data.session;
}
```

### Key gotchas

1. **Anonymous auth is DISABLED by default in Supabase projects.** You must turn it on in the dashboard: **Authentication → Sign In / Providers → Anonymous Sign-Ins → Enable**. This is a manual step; the plan must list it explicitly. Without it, `signInAnonymously()` returns an error and nothing else works. **HIGH confidence, common first-time failure.**

2. **Session ready before first query.** Gate the rest of the app on an `AuthGate` component that calls `ensureSession()` once, shows a blank/loading state until it resolves, then renders the router. Do not race queries against an unready session.

3. **`persistSession: true` writes to localStorage automatically** under a key like `sb-<project-ref>-auth-token`. Do NOT manually write the JWT — let the client own it.

4. **Anonymous users count toward Supabase MAU.** REQUIREMENTS.md already flags this in the Cost Risk Register. Expected v1 volume is tiny (friends-and-family), so not a concern now. Worth a footnote in the plan.

5. **`auth.uid()` in RLS resolves to the anon user's UUID** — it is a real UUID, stable for the life of the session, just like a signed-in user. You can use it as `session_id` without any special handling.

6. **Anonymous sessions can be upgraded later** (Phase 3's "optional account creation") via `supabase.auth.updateUser({ email })` — identity-linking preserves the `auth.uid()`. Not this phase, but worth knowing the flow doesn't throw away data.

**Confidence:** HIGH (official docs + widely documented).

## RLS Policy Shape for Insert-Only Tables

### The policy
```sql
-- question_plays
CREATE POLICY "anon can insert own plays"
  ON question_plays
  FOR INSERT
  TO anon, authenticated
  WITH CHECK (auth.uid() IS NOT NULL AND session_id = auth.uid());

-- Deliberately NO SELECT, UPDATE, DELETE policies.
-- With RLS enabled and no policy for an action, the action is denied.

ALTER TABLE question_plays ENABLE ROW LEVEL SECURITY;
```

Same pattern for `quiz_sessions`.

### The `.insert().select()` trap (critical)

When PostgREST inserts with a `RETURNING` clause (which `supabase-js` does by default when you chain `.select()` after `.insert()`), the returned rows are filtered through **SELECT RLS**. If there is no SELECT policy, the RETURNING rows come back empty and `supabase-js` reports the insert as failed (no row returned) — **even though the row was actually written**.

**Fix:** call `.insert(row)` **without** `.select()`. `supabase-js` will not issue `RETURNING`, PostgREST will not filter, and you get a clean success response.

```ts
// GOOD
const { error } = await supabase.from('question_plays').insert(row);

// BAD — returns empty + looks like failure under insert-only RLS
const { data, error } = await supabase.from('question_plays').insert(row).select();
```

The client doesn't need the generated `id` back — it already knows `session_id`, `question_id`, and the rest. This keeps the RLS surface minimal (zero SELECT on these tables).

**Confidence:** HIGH — this is a documented PostgREST/Supabase behaviour and the #1 gotcha on insert-only tables. The plan must call it out, and tests must assert insert works without `.select()` AND insert with `.select()` produces an empty-data result (or add a minimal SELECT-own-rows policy, which CONTEXT.md explicitly rejects).

### Anon vs authenticated role
An anonymous user in Supabase is in the `authenticated` role (not `anon` — confusingly). The `anon` role is for requests with no JWT at all. Since we always call `signInAnonymously()` before any insert, the role is `authenticated`. Include both in `TO anon, authenticated` to be safe — it won't hurt.

## Schema Recommendation

### `question_plays` (live per-question)
```sql
CREATE TABLE question_plays (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL,                    -- = auth.uid()
  question_id UUID NOT NULL REFERENCES questions(id) ON DELETE CASCADE,
  chosen_option TEXT NOT NULL,                 -- the text the user tapped
  is_correct BOOLEAN NOT NULL,
  time_to_answer_ms INTEGER NOT NULL CHECK (time_to_answer_ms >= 0),
  feedback_reaction TEXT CHECK (feedback_reaction IN ('good', 'bad', 'confusing')),
  played_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_question_plays_question ON question_plays(question_id);
CREATE INDEX idx_question_plays_played_at ON question_plays(played_at DESC);
-- No index on session_id — we never query by it (insert-only).
```

### `quiz_sessions` (one row per completed/abandoned quiz)
```sql
CREATE TABLE quiz_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL,                    -- = auth.uid()
  category_slug TEXT NOT NULL,                 -- 'general' | one of the 12 seed slugs
  difficulty TEXT NOT NULL CHECK (difficulty IN ('easy', 'normal', 'hard')),
  num_questions INTEGER NOT NULL CHECK (num_questions IN (5, 10, 15, 20)),
  score INTEGER NOT NULL CHECK (score >= 0),
  overall_rating TEXT CHECK (overall_rating IN ('good', 'okay', 'bad')),
  feedback_text TEXT,
  started_at TIMESTAMPTZ NOT NULL,
  completed_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_quiz_sessions_completed ON quiz_sessions(completed_at DESC);
CREATE INDEX idx_quiz_sessions_category ON quiz_sessions(category_slug);
```

### Notes
- **Enum vs TEXT+CHECK:** Use `TEXT` + `CHECK` (matches existing schema style in `00001_initial_schema.sql` which uses `TEXT CHECK` for `difficulty` and `status`). Don't introduce Postgres `ENUM` types — they are annoying to alter later.
- **No FK on `session_id` → `auth.users`.** Anonymous users exist in `auth.users` but referencing the auth schema from application tables is discouraged (Supabase docs advise against it). The insert-policy `session_id = auth.uid()` is the real integrity guarantee.
- **`category_slug` as text, not FK to `categories(slug)`.** Keeps the insert-only table light, and "general" is a virtual category with no row. If the planner prefers normalisation, add `category_id UUID NULL REFERENCES categories(id)` and treat NULL as General — both work.
- **Realtime:** explicitly DO NOT add these tables to the `supabase_realtime` publication. COST-04 and CONTEXT are clear on this.
- **`started_at` is client-supplied.** The client records when the user hit Play; server uses `now()` for `completed_at`. This is fine — we don't trust the client-supplied timestamp for anything important.

**Confidence:** HIGH on the shape, MEDIUM on a couple of column naming choices (planner can adjust).

## Question Fetch Strategy

### Recommendation: a Postgres function (RPC)

Create a SQL function that handles random sampling + optional category walk. Call it from the client with `supabase.rpc(...)`.

```sql
CREATE OR REPLACE FUNCTION random_published_questions(
  p_difficulty TEXT,           -- 'easy' | 'normal' | 'hard'
  p_category_slug TEXT,        -- NULL or 'general' for any; otherwise seed category slug
  p_limit INT                  -- 5 | 10 | 15 | 20
)
RETURNS TABLE (
  id UUID,
  question_text TEXT,
  correct_answer TEXT,
  distractors JSONB,
  explanation TEXT,
  category_id UUID
)
LANGUAGE SQL
STABLE
SECURITY INVOKER
SET search_path = public
AS $$
  WITH cat_scope AS (
    SELECT id FROM categories
    WHERE p_category_slug IS NULL
       OR p_category_slug = 'general'
       OR id IN (
         -- all descendants of the seed slug (including itself)
         WITH RECURSIVE tree AS (
           SELECT id FROM categories WHERE slug = p_category_slug
           UNION ALL
           SELECT c.id FROM categories c JOIN tree t ON c.parent_id = t.id
         )
         SELECT id FROM tree
       )
  )
  SELECT q.id, q.question_text, q.correct_answer, q.distractors, q.explanation, q.category_id
  FROM questions q
  WHERE q.status = 'published'
    AND q.difficulty = p_difficulty
    AND q.category_id IN (SELECT id FROM cat_scope)
  ORDER BY random()
  LIMIT p_limit;
$$;

-- Allow anonymous users to call it
GRANT EXECUTE ON FUNCTION random_published_questions(TEXT, TEXT, INT) TO anon, authenticated;
```

**Why RPC over PostgREST filter:**
- Keeps the recursive category walk on the server. Much cleaner than fetching every category and doing it in JS.
- One round-trip. Planner doesn't need to juggle two queries (fetch category tree, then fetch questions).
- `SECURITY INVOKER` means it runs with the caller's RLS — which for `questions` is the existing "published only" SELECT policy. No privilege escalation.

**Client call:**
```ts
const { data, error } = await supabase.rpc('random_published_questions', {
  p_difficulty: 'easy',
  p_category_slug: 'science',  // or null / 'general'
  p_limit: 10,
});
```

### `ORDER BY random() LIMIT N` at current scale
At 309 rows (growing to single-digit thousands in the near term) this is completely fine — Postgres scans the filtered set once and picks N. The scaling concern kicks in at ~100k rows per filter, where `TABLESAMPLE` or a precomputed shuffled column starts to matter. **Flag for later, don't solve now.**

### Building the 4-option array in the client
```ts
function buildOptions(row: { correct_answer: string; distractors: string[] }) {
  const all = [row.correct_answer, ...row.distractors];
  // Fisher-Yates or a seeded shuffle; for v1 Math.random() is fine.
  return shuffle(all);
}
```
Remember: the client needs to remember which of the four is correct, to judge `is_correct` when the user taps. Simplest: store `{ options: string[], correctIndex: number }` per question in quiz state.

**Confidence:** HIGH

## Active-Tab Timer Pattern

### Contract
Accumulate milliseconds only while **both** of these are true:
1. `document.visibilityState === 'visible'`
2. `document.hasFocus()`

Pause on `visibilitychange` → hidden, or `blur`. Resume on `visibilitychange` → visible, or `focus`, but only if the other condition is also true.

### Reference implementation

```ts
// apps/web/src/lib/activeTimer.ts
export function createActiveTimer() {
  let startedAt: number | null = null;  // ms epoch when currently running, else null
  let accumulated = 0;

  const isActive = () =>
    document.visibilityState === 'visible' && document.hasFocus();

  const start = () => {
    if (startedAt !== null) return;
    if (!isActive()) return;
    startedAt = performance.now();
  };

  const pause = () => {
    if (startedAt === null) return;
    accumulated += performance.now() - startedAt;
    startedAt = null;
  };

  const reset = () => {
    accumulated = 0;
    startedAt = isActive() ? performance.now() : null;
  };

  const elapsedMs = () => {
    const live = startedAt !== null ? performance.now() - startedAt : 0;
    return Math.round(accumulated + live);
  };

  const onVisibility = () => (isActive() ? start() : pause());
  const onFocus = () => (isActive() ? start() : pause());
  const onBlur = () => pause();

  document.addEventListener('visibilitychange', onVisibility);
  window.addEventListener('focus', onFocus);
  window.addEventListener('blur', onBlur);

  const destroy = () => {
    pause();
    document.removeEventListener('visibilitychange', onVisibility);
    window.removeEventListener('focus', onFocus);
    window.removeEventListener('blur', onBlur);
  };

  return { start, pause, reset, elapsedMs, destroy };
}
```

### Integration
- Create one timer per question. Call `reset()` when a new question is shown. Call `elapsedMs()` when the user taps an answer (NOT at reveal — the moment of answer selection is what you measure).
- Use `performance.now()` not `Date.now()` — monotonic, not affected by clock changes.
- Minimum the timer can report is 0. A user can answer in <50ms on a pre-loaded question; that's fine, let it through.

### Gotchas
- On mobile Safari, `document.hasFocus()` behaves oddly when the page is in an iframe — we're not in an iframe, non-issue for v1.
- iOS Safari pauses JS when the tab is backgrounded but fires `visibilitychange` on resume. Works correctly with the above.
- When the user takes a screenshot the tab briefly loses focus on some platforms. Unavoidable; the timer will pause for a few hundred ms. Acceptable.

**Confidence:** HIGH on the pattern, MEDIUM on the exact mobile-browser edge cases (untested in research — flag for a quick manual check during implementation).

## Outbox Pattern

### localStorage schema
Two keys, one per table:
```ts
const KEY_PLAYS = 'pub-quiz:outbox:question_plays:v1';
const KEY_SESSIONS = 'pub-quiz:outbox:quiz_sessions:v1';

type OutboxEntry<T> = {
  payload: T;
  enqueuedAt: number;  // ms epoch
};
```

Value is `JSON.stringify(OutboxEntry<Row>[])`. Cap at 500 entries; on overflow, drop oldest (shift from the front).

### Flush triggers
Keep it simple:
1. **On app start** (after `ensureSession()` resolves, before routing) — flush both outboxes once.
2. **After every successful insert** — try to drain the matching outbox. One extra await, no big deal.
3. **No interval timer.** No `beforeunload` flushes (unreliable, and data is already in localStorage so it survives anyway).

### Flush algorithm
```ts
async function flushOutbox<T>(key: string, insert: (row: T) => Promise<{ error: unknown }>) {
  const queue = loadQueue<T>(key);
  if (queue.length === 0) return;
  const remaining: OutboxEntry<T>[] = [];
  for (const entry of queue) {
    const { error } = await insert(entry.payload);
    if (error) {
      remaining.push(entry);  // keep it; try again next time
      break;                   // stop on first failure to preserve order
    }
  }
  saveQueue(key, remaining);
}
```

### Ordering guarantees
- **Within a table:** FIFO. The `break` on first failure preserves order.
- **Across tables:** none needed — `question_plays` and `quiz_sessions` have no FK relationship to each other.
- **Within `question_plays` for one quiz:** FIFO within the queue, but a quiz in progress inserts live, so the outbox only collects failed rows; they'll replay in the order they were attempted.

### Size cap
```ts
const MAX_OUTBOX = 500;
if (queue.length >= MAX_OUTBOX) queue.shift();  // drop oldest
queue.push(entry);
```
500 rows × a few hundred bytes each ≈ under 200 KB, well within the ~5 MB localStorage budget.

### `session_id` drift
If the anonymous JWT expires and a fresh anon sign-in assigns a new `auth.uid()`, queued rows with the old `session_id` will fail RLS forever. **Mitigation:** stamp outbox rows with the `session_id` captured at enqueue time (already implied), and during flush, if an RLS error is detected, drop the entry rather than retry forever. v1 keeps it simple — log and drop.

**Confidence:** HIGH

## State Management Choice

**Recommendation: `useReducer` for the quiz state machine, plain `useState` for the setup screen.**

### Why useReducer
The quiz state is a classic finite-state machine:
```
idle
  → loading (user hits Play)
  → playing(index)
    → revealed(index)
      → playing(index+1) | finished
```
Plus per-question data: chosen option, is_correct, elapsed ms, feedback reaction (nullable until tap).

`useReducer` gives you:
- A single source of truth — the reducer function IS the state machine.
- Trivially testable — pass state + action, assert next state. No React, no Supabase, no DOM.
- **Phase 7 lift is clean** — the reducer file moves into the native app as-is. React Native has the same `useReducer`, same React 19. Only the view layer changes.
- No new dependency. The "deliberately plain" ethos cashes in here.

### Why not Zustand
Zustand is great, but for a single-screen state machine with zero cross-component sharing beyond the play screen, it's ceremony. Phase 7 may want it for global app state (settings, theme) but that's a Phase 5 decision, not this phase's.

### Why not useState
`useState` scales poorly past ~3 related state variables. The quiz has: current index, answers array, phase (playing/revealed), start time, elapsed, feedback-in-flight flag. Six+ pieces. Reducer wins.

### Shape sketch
```ts
type QuizState =
  | { phase: 'idle' }
  | { phase: 'loading' }
  | {
      phase: 'playing' | 'revealed' | 'finished';
      questions: LoadedQuestion[];
      index: number;
      answers: AnswerRecord[];
      startedAt: number;
    };

type QuizAction =
  | { type: 'START'; questions: LoadedQuestion[] }
  | { type: 'ANSWER'; chosenIndex: number; elapsedMs: number }
  | { type: 'FEEDBACK'; reaction: 'good' | 'bad' | 'confusing' | null }
  | { type: 'NEXT' }
  | { type: 'RESET' };
```
Feedback tap dispatches `FEEDBACK` then `NEXT` (or the reducer handles both in one action — planner's call).

**Confidence:** HIGH on the recommendation. It's a judgement call but well-motivated by the Phase 7 lift goal.

## Routing Shape

**Recommendation: three routes, not a single-page state machine.**

```
/          → SetupScreen       (category, difficulty, count, Play button)
/play      → PlayScreen        (quiz state machine lives here)
/done      → EndScreen         (score, rating, feedback text, share)
```

Why separate routes:
- **Share link** needs query params on `/` — routes make that trivial (`useSearchParams`).
- **Back button** works naturally. From `/done`, back goes to `/play` which re-navigates to `/` (or a guard redirects if quiz state is gone).
- **Mental model** matches phase semantics.

### Query params on `/`
React Router v6/v7 `useSearchParams`:
```ts
const [params] = useSearchParams();
const cat = params.get('cat') ?? 'general';
const diff = params.get('diff') ?? 'easy';
const n = Number(params.get('n') ?? '10');
```
Validate: if `cat` not in the allow-list of 12 seed slugs + 'general', ignore. If `diff` not in `{easy,normal,hard}`, default. If `n` not in `{5,10,15,20}`, default.

### Navigating between routes
- Setup → `navigate('/play', { state: { questions, config } })` — pass loaded questions via router state. Avoids a second fetch on page mount.
- Play → Done: same pattern, `navigate('/done', { state: { summary, config } })`.
- If someone deep-links `/play` or `/done` directly, redirect to `/` (no state = no quiz to continue).

### Quiz state survives route change?
Only for the life of the route transition (router state is in memory). If the user refreshes `/play`, the quiz is gone and they go back to `/`. That is fine for v1 — resume-on-refresh is explicitly deferred.

**Confidence:** HIGH

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Anonymous session IDs | localStorage UUID | Supabase `signInAnonymously()` | Real JWT, RLS-enforceable, Phase 3 reuse, survives client code changes |
| JWT storage/refresh | Manual localStorage handling | `supabase-js` default `persistSession: true` | Handles refresh, expiry, cross-tab sync |
| Random question picker | Client-side shuffle of all published rows | Postgres `ORDER BY random() LIMIT N` in an RPC | One round-trip, no data leak of unreleased questions, scales to thousands |
| Category tree walk | Recursive JS walk over fetched categories | Postgres recursive CTE in the same RPC | One query, stays on the DB side |
| SPA fallback routing | Custom 404 handler | `public/_redirects` one-liner | Canonical Cloudflare Pages pattern |
| Insert-only row protection | App-level checks | Postgres RLS with no SELECT policy | Enforced at the database edge; app bugs can't leak data |
| State machine | Ad-hoc `useState` flags | `useReducer` with explicit states | Testable, Phase 7 portable, no dependency |
| Time tracking | `Date.now()` deltas | `performance.now()` + visibility/focus listeners | Monotonic, meets the active-tab contract |

**Key insight:** This phase's whole point is to be thin. Every "don't hand-roll" above also means "don't introduce a library for it" — the stack is Vite + React + React Router + supabase-js. That is the entire runtime dependency list the plan should aim for.

## Common Pitfalls

### Pitfall 1: Anonymous auth not enabled in Supabase project
**What goes wrong:** `signInAnonymously()` returns error `"Anonymous sign-ins are disabled"`. Whole app fails at load.
**Why it happens:** It's off by default in every new Supabase project and was introduced in 2024.
**How to avoid:** First task in the plan: "Enable Anonymous Sign-Ins in Supabase dashboard → Authentication → Providers." Add a smoke test in the PR review checklist.
**Warning signs:** Browser console shows auth error on page load, no session established.

### Pitfall 2: `.insert().select()` silently failing on insert-only RLS
**What goes wrong:** Insert succeeds in Postgres, but supabase-js returns `data: []` because RETURNING is filtered by a non-existent SELECT policy. The app treats it as a failure and queues to outbox, leading to duplicate rows on next successful insert.
**Why it happens:** Default ORM ergonomics — copy-paste examples all chain `.select()`.
**How to avoid:** Code review rule. Lint rule if you're keen (ESLint custom rule: ban `.insert().select()` on `question_plays` / `quiz_sessions`). At minimum, a unit test that asserts the insert helper does not chain `.select()`.
**Warning signs:** Rows appearing twice in the DB. Outbox never draining.

### Pitfall 3: `public` anon key in the wrong env var / committed to repo
**What goes wrong:** Anon key is public by design, but committing it in `.env` (not `.env.local`) leaks it into Git history unnecessarily. Worse: committing the service-role key.
**How to avoid:** `.gitignore` `apps/web/.env.local`. Never create `.env` in the web app. Set values in Cloudflare Pages dashboard only. Add a `git grep`/secret-scan step to the plan's validation.
**Warning signs:** Service-role key in `git log -p`. This is a full-project compromise.

### Pitfall 4: Timer keeps counting during reveal
**What goes wrong:** The user taps an answer, sees the reveal, reads the explanation for 20 seconds, taps feedback → `time_to_answer_ms` reports 22 seconds instead of 2.
**Why it happens:** Reducer never paused the timer at `ANSWER`.
**How to avoid:** `ANSWER` action reads `elapsedMs()` from the timer AND calls `pause()`. Reveal phase never reads the timer again. Unit test covers this explicitly.
**Warning signs:** Bimodal `time_to_answer_ms` distribution with a long tail in the 10–30s range.

### Pitfall 5: Outbox flush races with live insert
**What goes wrong:** User finishes a quiz, live insert for the last question succeeds, flush starts, then the user taps something that triggers another insert while flush is draining. Interleaved writes, possibly out of order.
**Why it happens:** Flush is async, state has moved on.
**How to avoid:** Single in-flight flush at a time. Use a module-level `isFlushing` flag. If a live insert starts while flush is running, queue it to the outbox directly (let the flush drain it). Simpler than a mutex.
**Warning signs:** Console errors about concurrent writes. Rows arriving out of chronological order.

### Pitfall 6: Category slug mismatch
**What goes wrong:** Landing page hardcodes `cat=science` but the seed category's slug is `science-and-nature`. RPC returns zero rows, user sees "no questions".
**How to avoid:** The 12 seed slugs + General list is hardcoded in **one** config module (`apps/web/src/config/categories.ts`), sourced directly from the existing `categories` table at build time OR manually mirrored with a comment pointing to the migration file. Planner picks one; either is fine for 12 rows.
**Warning signs:** Empty quiz, no error. Test: "every hardcoded slug returns ≥1 question at every difficulty."

### Pitfall 7: Difficulty label mismatch (`medium` vs `normal`)
**What goes wrong:** UI picker says "Medium", RPC query sends `'medium'`, DB has `'normal'`, zero rows returned.
**How to avoid:** Single translation function `uiToDbDifficulty('Medium') → 'normal'`. One place. Unit test with a table of all three.
**Warning signs:** "Medium" difficulty always returns empty.

### Ensure session on boot
```ts
// apps/web/src/lib/auth.ts
import { supabase } from './supabase';

let sessionPromise: Promise<string> | null = null;

export function ensureSessionId(): Promise<string> {
  if (!sessionPromise) {
    sessionPromise = (async () => {
      const { data: existing } = await supabase.auth.getSession();
      if (existing.session) return existing.session.user.id;
      const { data, error } = await supabase.auth.signInAnonymously();
      if (error) throw error;
      return data.session!.user.id;
    })();
  }
  return sessionPromise;
}
```

### Insert a question play (correct, no `.select()`)
```ts
// apps/web/src/lib/plays.ts
import { supabase } from './supabase';
import { enqueue, flushAll } from './outbox';

export type QuestionPlayRow = {
  session_id: string;
  question_id: string;
  chosen_option: string;
  is_correct: boolean;
  time_to_answer_ms: number;
  feedback_reaction: 'good' | 'bad' | 'confusing' | null;
  played_at: string;   // ISO
};

export async function recordQuestionPlay(row: QuestionPlayRow) {
  const { error } = await supabase.from('question_plays').insert(row);
  if (error) {
    enqueue('question_plays', row);
    return;
  }
  // Opportunistic drain
  flushAll().catch(() => { /* ignore */ });
}
```
