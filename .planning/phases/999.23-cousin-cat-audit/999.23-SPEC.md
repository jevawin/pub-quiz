Resolve all 19 cap-5 collision questions deferred from 999.22 and review every mis-tag candidate flagged by 999.22 backfill subagents (~30 Qs), via batched conversational review with user confirmation, applying changes through service-role SQL — without breaking the trigger cap or chain-row invariants from 999.22.

## Background

Phase 999.22 chain-tagged 1540 of 1559 published Qs. 19 Qs were skipped because adding chain ancestors would exceed the cap-5 row limit on `question_categories` (trigger `enforce_question_categories_rules` from migration `00034`). These Qs already carry 4–5 existing tags including over-applied cousins (e.g. V for Vendetta tagged at `action-heroes` + `classic-hollywood` + `british-history` + `general-knowledge` + `movies-and-tv`). Subagents also surfaced ~30 mis-tag candidates across batches: wide misuse of `90s-music-hits` on non-90s electronic acts (Aphex Twin, Madeon, AlunaGeorge), `classic-sitcoms` used as generic TV bucket (Black Mirror, Rick and Morty, M*A*S*H), `traditional-pub-games` on bowling Turkey/Monopoly, `world-cuisine` on Panama hats/Shiatsu/Spanish "donkey", a few near-duplicate Q pairs.

- `.planning/phases/999.22-chain-tagging-architecture/PROGRESS.md` — partial flagged list (4 Qs explicit; ~26 surfaced in subagent runs).

No code changes expected. Pure data audit + targeted DELETE/INSERT on `question_categories` (and possibly `questions.category_id` for primary moves).

## Requirements

1. **Cap-5 collision resolution**: Every one of the 19 deferred Qs gets to a state where the chain ancestors from 999.22 are present.
   - Current: 19 Qs in `batch-001.json` have chain ancestors missing because adding them would breach cap-5 (existing tags = 4–5).
   - Target: Each of the 19 Qs has its over-applied cousin tags pruned (DELETE on `question_categories`) so the missing chain ancestor rows can be inserted via the existing `apply-chain-rows.ts` script. Final row count per Q ≤ 5 and includes the full chain.
   - Acceptance: Re-running `pipeline/src/scripts/build-chain-backfill-worklist.ts` reports 0 Qs remaining. Each of the 19 Q ids now has its expected ancestors per `batch-001.json` reflected in `question_categories`.

2. **Mis-tag candidate review**: Every Q flagged by 999.22 subagents is reviewed and a decision committed.
   - Current: ~30 flagged Q ids exist (4 explicit in PROGRESS.md `Subagent observations to feed 999.23`; ~26 surfaced in subagent run output across batches 1–15). No decision recorded yet.
   - Target: A consolidated worklist file `data/mistag-worklist.json` captures every flagged Q id + reason + decision (`fix` / `keep` / `defer`). Every `fix` decision is applied to `question_categories` (and `questions.category_id` when primary cat is wrong).
   - Acceptance: `mistag-worklist.json` exists with one entry per flagged Q. Every entry has a `decision` field set. Every `fix` entry has a corresponding row mutation in the audit log file `data/audit-changes.jsonl` (one line per DB write).

3. **Strict cousin rule enforced**: New cousin tags added during this phase obey the ROADMAP rule.
   - Current: No cousin curation pass has run; subagent backfill explicitly skipped cousins.
   - Target: Any cousin tag added must satisfy: "a player picking that cat would genuinely expect this Q." Use `pop-culture` as cousin for MCU / Disney / Star Wars / mainstream gaming franchise Qs as opportunity arises in the cap-5 + mis-tag worklist. Never add `general-knowledge` as a cousin to a chain-tagged geography/history/etc Q purely because it is well-known.
   - Acceptance: `audit-changes.jsonl` rows that add a cousin (slug not in chain ancestors) include a `cousin_reason` field with a one-sentence justification matching the strict rule.

4. **Cap-5 invariant holds throughout**: No Q ends the phase with > 5 `question_categories` rows.
   - Current: Trigger `enforce_question_categories_rules` enforces cap-5 (mig 00034). Trigger blocks any INSERT that would exceed it.
   - Target: Phase ships with the trigger still in place and zero violations. All deletes happen before inserts so ordering never trips the trigger.
   - Acceptance: `SELECT question_id, COUNT(*) FROM question_categories GROUP BY question_id HAVING COUNT(*) > 5` returns 0 rows post-phase.

5. **Format = batched chat review with confirmation**: User-in-the-loop is the execution model.
   - Current: No tool exists for this workflow. 999.22 used subagent dispatch (autonomous); 999.23 is explicitly conversational per ROADMAP.
   - Target: Work proceeds in batches of ~10–30 Qs presented as a markdown table (Q text, current tags, proposed adds/removes, reasoning) in the chat. User responds with confirmations / edits / rejects per row. Approved rows applied via service-role SQL (Supabase MCP or pipeline script). Each batch commits atomically with audit-log entries.
   - Acceptance: At least one commit per batch in `git log` referencing 999.23 with a batch number and Q count. Every commit's diff includes appended rows in `data/audit-changes.jsonl`.

## Boundaries

**In scope:**
- Resolving the 19 cap-5 Qs from `batch-001.json`.
- Reviewing + acting on the ~30 mis-tag candidates flagged by 999.22 subagents.
- Adding `pop-culture` cousin tags where strict rule is satisfied (MCU, Disney, Star Wars, mainstream gaming franchises encountered in the worklist).
- Moving primary `questions.category_id` where mis-categorised (e.g. Aladdin Q at `literature` → `movies-and-tv`).
- Pruning over-applied cousin tags to free cap-5 headroom.
- Atomic commits per batch with audit log.

**Out of scope:**
- Re-running chain calibration — chain rows from 999.22 are correct; this phase only touches non-chain (cousin) rows + primary cat moves.
- Broad audit of all 2848 published Qs — only the surfaced 19 + ~30 + opportunistic cousins encountered in those batches.
- Code changes to calibrator, RPCs, or trigger — schema and pipeline are frozen.
- Generating new questions or content rewrites — content unchanged; only categorisation moves.
- Removing or re-classifying near-duplicate Q pairs flagged by 999.22 — defer to a separate dedup quick task (not this phase).
- Bulk SQL migrations — all changes via service-role calls captured in audit log; no new files in `supabase/migrations/`.
- Mis-categorised question_text / answer corrections — content quality is a separate workflow (`question_feedback`).

## Constraints

- Trigger `enforce_question_categories_rules` from `00034` enforces cap-5; sequence DELETEs before INSERTs per Q.
- Chain ancestor rows added by 999.22 are immutable — never delete a chain ancestor row to make room for cousins.
- Use service-role for writes (Supabase MCP or pipeline script with `SUPABASE_SERVICE_ROLE_KEY`); RLS would block this.
- All writes append a JSONL line to `data/audit-changes.jsonl` (Q id, slug, op, reason, timestamp) — non-negotiable for traceability.
- Strict cousin rule per ROADMAP §999.23 — cousin only when a player picking that cat would genuinely expect this Q.
