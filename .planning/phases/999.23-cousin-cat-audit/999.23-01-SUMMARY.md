# Phase 999.23 Plan 01: apply-cousin-changes script ‚Äî Summary

**One-liner:** Service-role TypeScript script that applies delete/insert/set_primary ops to question_categories with deletes-before-inserts ordering, mandatory cousin_reason on non-chain inserts, and per-op JSONL audit trail ‚Äî wired with offline-safe `--dry-run` and 5 vitest cases (TDD).

## What was built

Two artifacts ready for waves 2-5 to consume:

1. **`pipeline/src/scripts/apply-cousin-changes.ts`** (439 lines) ‚Äî mirrors `apply-chain-rows.ts` shape (shebang, dotenv override, `createSupabaseClient`, async `run()`, exit-1 on validation failure). Key differences from the reference script:
   - Input is a wrapping object `{ batch_id, ops: ChangeOp[] }` so `batch_id` stamps every audit line.
   - Discriminated `ChangeOp` union ‚Äî `delete | insert | set_primary` ‚Äî validated up front (uuid shape on `question_id`, `estimate_score` in `[0,100]`, mandatory fields per op type).
   - Per-question grouping (`Map<question_id, GroupedOps>`) with strict execution order: ALL deletes ‚Üí ALL inserts ‚Üí ALL set_primary. This satisfies the cap-5 trigger from migration `00034` even when an insert would briefly breach the cap before its paired delete completes.
   - **D5 enforcement**: insert ops with `chain_ancestor !== true` MUST carry a non-empty `cousin_reason`; payload validation rejects non-conforming ops with stderr listing qid + slug, exits 1.
   - `prev_score` per delete is fetched in one batched query per Q (`select('category_id, estimate_score').eq('question_id').in('category_id', ...)`) before any delete fires, so the audit log captures the score being pruned.
   - Idempotent inserts via `upsert` + `ignoreDuplicates: true` on conflict `question_id,category_id` ‚Äî re-running a batch is a no-op.
   - Synchronous `fs.appendFileSync` to `data/audit-changes.jsonl` (relative to `process.cwd()`) immediately after each successful DB op; ordering of the JSONL file matches operation order.
   - `--dry-run` flag short-circuits BEFORE Supabase client construction ‚Äî no env vars required, no slug-resolution query, no audit-log writes. Returns plan grouped by question_id.

2. **`pipeline/src/scripts/apply-cousin-changes.test.ts`** (329 lines, 5 cases via vitest):
   - rejects insert without cousin_reason when `chain_ancestor !== true`
   - `--dry-run` does not invoke supabase write methods (mock asserts)
   - ops are grouped by question_id with deletes-before-inserts ordering (call-order assertion)
   - audit log JSONL line written per successful op (`fs.appendFileSync` called 3√ó for delete+insert+set_primary; delete entry records `prev_score` from the pre-fetch)
   - missing slug in categories table ‚Üí exit 1 with the slug listed

3. **`.planning/phases/999.23-cousin-cat-audit/999.23-smoke-payload.json`** ‚Äî exercises all three op types against one real cap-5 Q (`6a535027-b95e-4734-a9be-4b9c7e5dacd3`, "V for Vendetta") sourced from `.planning/phases/999.22-chain-tagging-architecture/data/batches/batch-001.json`. Used for regression smoke when waves 2-5 modify the script.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 ‚Äî Blocking issue] Dry-run required Supabase env vars**

- **Found during:** Task 2 smoke test from repo root (no `.env` at repo root, only at `pipeline/.env`).
- **Issue:** Original implementation constructed the Supabase client and ran the slug-resolution query before checking the `--dry-run` flag, so dry-run failed without env vars. This contradicted the plan intent ("`--dry-run` flag ‚Ä¶ do NOT write to DB, do NOT append to audit log") and the Task-2 acceptance criterion that dry-run exits 0.
- **Fix:** Moved the dry-run short-circuit to immediately after payload validation + grouping, before any Supabase or fs touch. Slug resolution + client construction now happen only on the real apply path.
- **Files modified:** `pipeline/src/scripts/apply-cousin-changes.ts`
- **Commit:** `3618622`
- **Tests:** still 5/5 green post-refactor.

- **Found during:** First `npx vitest run` attempt.

## Threat surface

No new surface beyond the threat model already in the plan. The script preserves the same posture as `apply-chain-rows.ts` (service-role write context behind stdin validation). One mitigation worth flagging for downstream waves:

- **T-99923-03 (audit trail repudiation)** is mitigated by `appendFileSync` AFTER each successful DB op. Note: if `appendFileSync` throws (disk full, permission), the script propagates the throw and exits non-zero ‚Äî the prior DB write is already committed (PostgreSQL autocommit). The plan accepts this trade-off per D5 ("JSONL is best-effort but per-op"). Operators MUST NOT rerun a batch on appendFileSync failure without first inspecting `audit-changes.jsonl` for the partial trail.
