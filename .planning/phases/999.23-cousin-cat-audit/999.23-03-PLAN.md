<interfaces>
Per-Q workflow (D4 prune heuristic):
1. Read Q from batch-001.json: `{question_id, question_text, existing_slugs[], chain_ancestors_needed[], scores[]}`.
2. Compute drop candidates: existing slugs that are NEITHER in chain_ancestors_needed NOR `general-knowledge`. Sort ascending by estimate_score.
3. Drop lowest-score candidate; if still over cap-5 after planned inserts, drop next-lowest.
4. NEVER drop a chain ancestor.
5. Drop `general-knowledge` only if estimate_score < 20.

Batch presentation table (D3):

| # | Q (≤80 chars) | Current slugs (score) | Drop | Insert (chain) | Net rows | Reason |
|---|---|---|---|---|---|---|

User responses:
- `all ok` / `all approved` → apply every row in the batch as proposed
- `rows 1,3,5 only` → apply listed rows; skip others (mark deferred)
- `row 2: drop X instead of Y` → edit per-row before apply
- `skip row N` → leave that Q unresolved (logged as deferred in batch)

Cap-5 trigger ordering (SPEC + D4): for each Q, ChangeOps array MUST list deletes before inserts. apply-cousin-changes.ts enforces this per question_id grouping.
</interfaces>
</context>
