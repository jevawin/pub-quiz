# Phase 999.8: Multi-Category + Per-Category Percentage Difficulty — Research

**Researched:** 2026-04-24
**Domain:** Postgres schema migration + pipeline agent rewrite + RPC + web client
**Confidence:** HIGH (all decisions locked in CONTEXT.md; codebase paths fully traced)

## Summary

This phase replaces three columns on `questions` (`category_id`, `difficulty`, `calibration_percent`) with a `question_categories` join table that carries `(estimate_score, observed_score, observed_n)` per (question, category) pair. Difficulty becomes a runtime band over the score. Every question must carry a `general-knowledge` row, enforced by a DB trigger plus pipeline guard.

Migration is a hard cutover: create the join table, re-run a rewritten Calibrator over all 2956 published questions to populate scores, then drop the old columns and update RPCs. Phase 2.2 plans 02–08 are blocked until this lands.

The bulk of the work is mechanical (write migration SQL, rewrite two agents, update three RPCs, update web client). The non-mechanical risks are: (1) ordering — RPCs and web client must read the new schema in the same deploy that drops old columns, or production goes 500; (2) data loss on the Calibrator re-run if it errors mid-batch; (3) subtle behavioural change as multi-category questions show up in more category quizzes than they did before.

**Primary recommendation:** Cutover in three migrations (create join + trigger; backfill via Calibrator; drop old columns + new RPCs + new agent code in same deploy). Gate the drop-columns migration behind a verification query confirming every published question has a `general-knowledge` row.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Schema — Old Columns**
- **D-01:** Drop `category_id`, `difficulty`, `calibration_percent` from `questions`. Clean break, no shims.

**Schema — `question_categories` Join Table**
- **D-02:** Shape: `question_categories(question_id UUID FK, category_id UUID FK, estimate_score NUMERIC(5,2), observed_score NUMERIC(5,2) NULL, observed_n INTEGER NOT NULL DEFAULT 0)`. PK = `(question_id, category_id)`.
- **D-03:** DB trigger enforces every `question_id` has at least one row where the referenced category slug = `'general-knowledge'`.
- **D-04:** DB trigger enforces max 4 rows per `question_id` (3 non-GK + 1 GK). App truncates to top-3 non-GK before write.

**Schema — Difficulty Tiers**
- **D-05:** No difficulty enum. Runtime band: hard 0–33, medium 34–66, easy 67–100. Thresholds in single config constant.

**Migration Path**
- **D-06:** Sequence: (1) create `question_categories` + constraints; (2) Calibrator re-run populates all 2956 published questions; (3) drop the three columns from `questions`; (4) update RPCs and query layer.
- **D-07:** Phase 2.2 plans 02–08 ship against new schema only. 999.8 must complete before 2.2 resumes.

**Agent Rewrites — Questions Agent**
- **D-08:** Questions Agent returns category slug array (e.g. `["science-and-nature", "general-knowledge"]`). Capped at 1–3 non-GK + mandatory GK. Pipeline resolves slugs to UUIDs.
- **D-09:** Prompt: always include `"general-knowledge"`, propose 1–3 specific extras, cap at 3. Returns per-category `estimate_score` in the same call.

**Agent Rewrites — Calibrator**
- **D-10:** One Haiku call per question. Inputs: question text, correct answer, distractors, assigned slug list. Output: JSON object keyed by snake_case slug + `"general_knowledge"`, each value 0–100. Prompt frames the audience as "% of players who *chose to play this category* who'd get it right".
- **D-11:** Migration re-run discards existing `calibration_percent` values — old prompt calibrated to a different audience.

**General Knowledge Enforcement**
- **D-12:** App guard validates slug array before DB write. Missing GK → log + skip (no insert). Trigger is the hard backstop.
- **D-13:** `'general-knowledge'` is a protected pipeline constant; Questions Agent cannot propose it as an "extra".

**Observed Score Refresh**
- **D-14:** Nightly cron aggregates `question_plays` grouped by `(question_id, category_id)`. Computes `observed_score = correct/total*100` and `observed_n = total`. Upserts to `question_categories`.
- **D-15:** Quiz selection RPC uses `CASE WHEN observed_n >= 30 THEN observed_score ELSE estimate_score END` as effective score. Computed in RPC, not a generated column.

### Trigger pattern
Use AFTER INSERT/UPDATE/DELETE row trigger that re-validates the question's row set. Standard PL/pgSQL:

```sql
CREATE OR REPLACE FUNCTION enforce_question_categories_rules()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  qid UUID := COALESCE(NEW.question_id, OLD.question_id);
  has_gk BOOLEAN;
  row_count INT;
BEGIN
  SELECT EXISTS (
    SELECT 1 FROM question_categories qc
    JOIN categories c ON c.id = qc.category_id
    WHERE qc.question_id = qid AND c.slug = 'general-knowledge'
  ) INTO has_gk;

  SELECT COUNT(*) FROM question_categories WHERE question_id = qid INTO row_count;

  IF row_count > 0 AND NOT has_gk THEN
    RAISE EXCEPTION 'Question % must have a general-knowledge row', qid;
  END IF;
  IF row_count > 4 THEN
    RAISE EXCEPTION 'Question % cannot have more than 4 categories (got %)', qid, row_count;
  END IF;
  RETURN NULL;
END $$;

CREATE CONSTRAINT TRIGGER trg_question_categories_rules
  AFTER INSERT OR UPDATE OR DELETE ON question_categories
  DEFERRABLE INITIALLY DEFERRED
  FOR EACH ROW EXECUTE FUNCTION enforce_question_categories_rules();
```

`DEFERRABLE INITIALLY DEFERRED` is essential — it lets the pipeline insert all 2-4 rows for a question in one transaction without the trigger firing on the first incomplete-set insert.

### RPC pattern (computed effective score)

```sql
-- Snippet inside random_published_questions rewrite
WITH effective AS (
  SELECT qc.question_id, qc.category_id,
         CASE WHEN qc.observed_n >= 30 THEN qc.observed_score ELSE qc.estimate_score END AS score
  FROM question_categories qc
)
SELECT q.id, q.question_text, q.correct_answer, q.distractors, q.explanation,
       e.category_id, c.slug AS category_slug
FROM questions q
JOIN effective e ON e.question_id = q.id
JOIN categories c ON c.id = e.category_id
WHERE q.status = 'published'
  AND e.score BETWEEN p_score_min AND p_score_max
  AND (p_category_slug IS NULL OR p_category_slug = 'general'
       OR e.category_id IN (SELECT id FROM cat_tree))
ORDER BY random()
LIMIT p_limit;
```

The RPC signature changes: `p_difficulty TEXT` becomes `p_score_min NUMERIC, p_score_max NUMERIC`. Web client computes the bounds from UI difficulty using the band constants (33/66) before the call. Mixed → 0/100.

## Common Pitfalls

### Pitfall 1: Trigger fires before transaction completes
**What goes wrong:** Inserting 4 rows for a question in 4 separate `INSERT` statements within one transaction — without `DEFERRABLE INITIALLY DEFERRED`, the trigger fires after row 1 (no GK yet) and aborts.
**How to avoid:** Always declare the constraint trigger as `DEFERRABLE INITIALLY DEFERRED`. Wrap the multi-row insert in an explicit transaction.
**Warning sign:** "Question X must have a general-knowledge row" raised on a write that includes a GK row.

### Pitfall 2: RPC and column drop in different deploys
**What goes wrong:** Drop `questions.difficulty` before redeploying RPCs that reference it → all quiz fetches return 500.
**How to avoid:** Migrations 00024 (drop columns) and 00025 (rewrite RPCs) ship together. Run them as a single Supabase migration push, not two separate ones.
**Warning sign:** Web client suddenly throws "column does not exist" errors.

### Pitfall 3: Backfill partial completion
**What goes wrong:** Calibrator re-run errors at question 1500/2956 due to budget cap or rate limit. Half the library has no `question_categories` rows. Drop-columns migration runs anyway → those questions become invisible.
**How to avoid:** Backfill script must be resumable (skip questions that already have ≥1 row in `question_categories`). Drop-columns migration must include a precondition check:
```sql
DO $$
DECLARE missing INT;
BEGIN
  SELECT COUNT(*) FROM questions q
  WHERE q.status = 'published'
    AND NOT EXISTS (SELECT 1 FROM question_categories qc WHERE qc.question_id = q.id)
  INTO missing;
  IF missing > 0 THEN
    RAISE EXCEPTION '% published questions still missing question_categories rows', missing;
  END IF;
END $$;
```

### Pitfall 4: Slug format mismatch (snake_case in JSON, kebab-case in DB)
**What goes wrong:** Calibrator returns `{"science_and_nature": 68}` but `categories.slug` is `science-and-nature`. Naive lookup returns null, score is dropped silently.
**How to avoid:** Canonical conversion helper — `jsonKeyToSlug(key) = key.replace(/_/g, '-')`. Validate every score key resolves to a known slug before write; throw if not.
**Warning sign:** Questions land with fewer category rows than the agent assigned.

### Pitfall 5: Multi-category surfaces questions in unexpected categories
**What goes wrong:** A "Pixar films that won the Best Animated Feature Oscar" question previously appeared only under `movies-and-tv/pixar`. Post-migration it has rows for `pixar`, `movies-and-tv`, and `general-knowledge`, so it surfaces in three quizzes — and Setup screen counts go up. Looks like a duplicate-question bug.
**How to avoid:** Update Setup screen pool counts to use `DISTINCT q.id` when joining `question_categories` so a question filed under N categories counts once per category-tree query but only once in totals. Document this in the agent prompt: "Only add a category if a player choosing that category would expect to see this question — not just if it's tangentially related."
**Warning sign:** Total question counts inflate vs pre-migration baselines.

### Pitfall 6: `question_plays` only carries `question_id`, not category context
**What goes wrong:** D-14's nightly aggregation needs `(question_id, category_id)` granularity, but `question_plays` only knows `question_id`. The same play counts toward every category that question is filed under.
**How to avoid:** This is correct and intentional — a person who chose "Pixar" and got the question right contributes to the Pixar `observed_score` for that question. The nightly job joins `question_plays → question_categories` and aggregates per `(question_id, category_id)`. Document this in the cron's SQL comments so a future reader doesn't try to "fix" it.
**Note:** `quiz_sessions.category_slug` records the player's chosen category — the join can be tightened later (Phase 2.2+) to count plays only against the actual chosen category. For 999.8, the simpler full-cross aggregation is fine; flag it as a known approximation in the cron job comments.

### Pitfall 7: NUMERIC(5,2) overflow
**What goes wrong:** `NUMERIC(5,2)` allows values up to 999.99 — fine — but a buggy Calibrator returning 100.5 inserts as 100.50, no error. Later a band check `<= 100` may pass or fail unexpectedly.
**How to avoid:** Add `CHECK (estimate_score BETWEEN 0 AND 100)` and `CHECK (observed_score IS NULL OR observed_score BETWEEN 0 AND 100)` on the join table.

## Code Examples

### Slug → UUID resolution before insert (pipeline)
```typescript
// pipeline/src/lib/categories.ts (new helper)
export async function resolveSlugsToIds(
  supabase: SupabaseClient<Database>,
  slugs: string[],
): Promise<Map<string, string>> {
  const { data, error } = await supabase
    .from('categories').select('id, slug').in('slug', slugs);
  if (error) throw error;
  const map = new Map<string, string>();
  for (const row of data ?? []) map.set(row.slug, row.id);
  for (const slug of slugs) {
    if (!map.has(slug)) throw new Error(`Unknown category slug: ${slug}`);
  }
  return map;
}
```

const BANDS: Record<Exclude<UiDifficulty, 'Mixed'>, { min: number; max: number }> = {

## Open Questions

1. **Should the nightly observed-score job run inside the existing pipeline workflow or as a separate one?**

   - What's unclear: Whether co-locating risks coupling (e.g. Calibrator budget cap aborts the observed-score refresh).
   - Recommendation: New separate GH Actions workflow `observed-score-refresh.yml`. No LLM calls, just a SQL-only Supabase RPC. Independent failure mode, smaller blast radius.

2. **Does the QA Agent stay difficulty-aware?**
   - What we know: QA currently has `difficulty_calibration_score` and `recalibrated_difficulty` fields. After this phase, there's no `difficulty` to recalibrate — the score per category is the source of truth.
   - What's unclear: Whether QA should now adjust per-category scores, or whether the Calibrator is the sole owner of scoring.
   - Recommendation: Strip difficulty fields from QA in this phase (cosmetic — set them to optional in the Zod schema and ignore them on write). A follow-up phase decides whether QA gets a "score audit" role. Don't expand scope here.

   - What we know: Bulk-import staging table still has a single `difficulty` column and `category_id`. Future OpenTDB imports promote rows from there into `questions`.
