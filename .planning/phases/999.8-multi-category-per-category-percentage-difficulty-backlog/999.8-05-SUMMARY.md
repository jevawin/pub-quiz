# Phase 999.8 Plan 05: Drop Legacy Columns + Rewrite RPCs Summary

**Cutover plan: legacy questions.category_id/difficulty/calibration columns dropped; all five affected RPCs rewritten to filter on question_categories effective scores; web client uses score-range UI buckets while keeping its public API stable.**

## Performance

- Started: 2026-05-04
- Completed: 2026-05-04 (Tasks 1+2; Task 3 is the blocking human-verify checkpoint)
- Tasks executed: 2 of 3 (Task 3 awaits manual smoke test)
- Files created: 2 SQL migrations + 1 deferred-items.md
- Files modified: 17 (5 web, 12 pipeline)

## Accomplishments

- Migration 00026 drops questions.category_id, questions.difficulty, questions.calibration_percent, questions.calibrated_at, plus the three legacy single-column indexes. Aborts with `RAISE EXCEPTION` if any published question still lacks a question_categories row.
- Migration 00027 rewrites five RPCs that previously read q.difficulty/q.category_id directly: `random_published_questions_excluding`, `count_available_questions`, `counts_by_root_category`, `random_general_knowledge_questions`, `random_published_questions`. All compute the effective score from question_categories with `CASE WHEN observed_n >= 30 THEN observed_score ELSE estimate_score END` and accept caller-supplied score ranges. All five `GRANT EXECUTE` to anon + authenticated.
- Web client gains `uiToScoreRange(UiDifficulty) → {min, max}`. Internal RPC calls switched to score-range params. Public function signatures unchanged so Setup/Play/End screens did not need to be touched.
- Pipeline updates: removed transitional category_id/difficulty placeholder writes from Questions Agent insert. Switched per-category counting and dedup queries (Questions Agent, category-selection, seed-threshold-check) onto question_categories. QA agent now joins question_categories → categories(name) for the prompt context (multiple category names, joined). Enrichment agent dropped the now-meaningless `Category: ${question.category_id}` prompt line. Calibrator's "needs calibration" filter switched from `calibrated_at IS NULL` to "no question_categories row exists".
- pipeline/src/lib/database.types.ts: dropped the four legacy columns from questions Row/Insert/Update + the questions_category_id_fkey relation; added a Functions block with typed signatures for the five rewritten RPCs.
- Test suites updated: pipeline tests/lib/category-selection.test.ts and tests/seed-threshold-check.test.ts mocks now branch on table === 'question_categories'. tests/agents/questions.test.ts mock updated to provide the new join-table query shape for dedup. apps/web tests rewritten for the new RPC contract.

## Decisions Made

- **Renumbered migrations to 00026 + 00027.** The plan was written assuming migration slots 00023/00024 were free, but quick-task migrations between phase planning and Plan 05 execution had taken 00023 (general-knowledge optional rule), 00024 (fun_fact in RPC return), and 00025 (sport category filter join). Migrations are committed in linear file order, so the new pair must come after 00025.
- **Preserved web client public API.** The plan's proposed shape was `fetchRandomQuestions({ categorySlug, scoreMin, scoreMax, limit })`, but Setup.tsx and Play.tsx call `fetchRandomQuestions(uiDifficulty, slugs, n)` and `countAvailableQuestions(uiDifficulty, slugs, bool)`. Changing the public signatures would have rippled into 3 screens + test files. Internal translation via `uiToScoreRange` keeps the change surgical.
- **CategoryCounts kept its `{ easy, normal, hard, total }` shape.** The new `counts_by_root_category` RPC returns rows with `difficulty: 'easy' | 'normal' | 'hard'` derived from score bands. Setup.tsx's `countForSlug` helper still works unchanged.
- **Stubbed `investigate-sport-leak.ts`.** This was a one-shot diagnostic that compared the legacy `questions.category_id` against `question_categories`. With the legacy column dropped, the comparison is meaningless. Replaced with a clear "OBSOLETE" log line.
- **Calibrator's `.is('calibrated_at', null)` → fetch-then-set-difference.** PostgREST has no clean "where no related row exists" predicate. Instead the calibrator fetches up to 2000 candidate questions, pulls their question_id presence from question_categories, and filters client-side to those without a row.
