# Phase 999.8: Multi-Category + Per-Category Percentage Difficulty — Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the single `category_id` + `difficulty` enum + `calibration_percent` columns on `questions` with a `question_categories` join table where each row carries `estimate_score`, `observed_score`, and `observed_n`. Every question gets a mandatory `general_knowledge` score entry. Difficulty tiers become a runtime band lookup over the score (hard 0–33, medium 34–66, easy 67–100) with no enum in the schema. Agents are rewritten to produce per-category scores. All 2956 existing published questions are migrated via a Calibrator re-run.

Phase 999.8 is a **prerequisite** for resuming Phase 2.2 — the remaining 2.2 plans (02–08) must target the new schema.

</domain>

<decisions>
## Implementation Decisions

### Schema — Old Columns

- **D-01:** Drop all three old columns from `questions`: `category_id`, `difficulty`, and `calibration_percent`. Clean break — `question_categories` is the sole source of truth. No backward-compat shims.

### Schema — question_categories Join Table

- **D-02:** Join table shape: `question_categories(question_id UUID FK, category_id UUID FK, estimate_score NUMERIC(5,2), observed_score NUMERIC(5,2) NULL, observed_n INTEGER NOT NULL DEFAULT 0)`. Primary key is `(question_id, category_id)`.
- **D-03:** DB trigger enforces that every `question_id` in `question_categories` has at least one row where the referenced category slug = `'general-knowledge'`. A question without a GK row is invalid.
- **D-04:** DB trigger enforces max 4 rows per `question_id` (up to 3 non-GK categories + 1 mandatory GK). Application code silently truncates to the 3 most relevant non-GK categories before writing — the DB trigger is the hard floor.

### Schema — Difficulty Tiers

- **D-05:** No difficulty enum anywhere. Runtime band lookup: `hard` = score 0–33, `medium` = 34–66, `easy` = 67–100. Thresholds live in a single config constant; tunable without touching data.

### Migration Path

- **D-06:** Migration executes in sequence: (1) create `question_categories` table with constraints; (2) run Calibrator re-run against all 2956 published questions, populating `question_categories` rows; (3) once all questions have GK rows, drop `category_id`, `difficulty`, `calibration_percent` from `questions`; (4) update all RPCs and query layer to read from `question_categories`.
- **D-07:** Phase 2.2 remaining plans (02–08) are updated to use the new schema. They do not ship against the old schema. 999.8 must complete before 2.2 resumes.

### Agent Rewrites — Questions Agent

- **D-08:** Questions Agent returns a JSON array of category slugs for each question (e.g. `["science-and-nature", "general-knowledge"]`). Capped at 1–3 non-GK slugs + mandatory `"general-knowledge"`. Pipeline resolves slugs to UUIDs via a categories table lookup — agent never handles UUIDs.
- **D-09:** Agent prompt instructs: always include `"general-knowledge"`, propose 1–3 additional categories that are genuinely specific (not padding), cap at 3 extra. Per-category `estimate_score` values are also returned in the same call (see D-10).

### Agent Rewrites — Calibrator

- **D-10:** One Calibrator (Haiku) call per question. Receives: question text, correct answer, distractors, and the list of assigned category slugs. Returns a JSON object with one key per slug + `"general_knowledge"`: `{ "general_knowledge": 45, "science_and_nature": 68 }`. Prompt asks it to reason about each audience separately: "% of players who *chose to play this category* who'd get it right".
- **D-11:** Migration re-run: Calibrator processes all 2956 published questions with the new prompt. Existing `calibration_percent` values are discarded — not used as seeds, because the old prompt calibrated to a different audience definition.

### General Knowledge Enforcement

- **D-12:** Application guard validates the slug array before any DB write. If `"general-knowledge"` is absent, the pipeline logs an error and skips the question (does not insert). DB trigger is the hard backstop for direct writes.
- **D-13:** `'general-knowledge'` category slug is a protected constant in the pipeline config — cannot be proposed as an "extra" category (it's always the mandatory baseline).

### Observed Score Refresh

- **D-14:** Nightly cron job (existing pipeline runner or new scheduled step) aggregates `question_plays` grouped by `(question_id, category_id)`, computes `observed_score = correct_count / total_count * 100` and `observed_n = total_count`, then upserts into `question_categories`. Runs once per day.
- **D-15:** Quiz selection RPC uses: `CASE WHEN observed_n >= 30 THEN observed_score ELSE estimate_score END` as the effective score for filtering and ranking. Threshold of 30 matches the roadmap specification. This is a computed expression in the RPC, not a generated column (avoids migration complexity).

### Claude's Discretion

- Exact column types and nullability beyond what's specified (e.g. whether `estimate_score` can be NULL on insert for questions mid-generation)
- Index design for `question_categories` (at minimum: index on `question_id`, index on `category_id`)
- Whether the nightly observed-score refresh runs as a new pipeline agent file or as an extension of an existing step
- Exact trigger implementation (PL/pgSQL vs constraint trigger)
- RPC names and signatures for the updated question selection queries

</decisions>

## Existing Code Insights

### Reusable Assets
- `pipeline/src/agents/calibrator.ts` — Current single-score Calibrator is the base to rewrite from. Same Haiku model, same token accumulator pattern, same Supabase update pattern — all reusable.
- `pipeline/src/agents/questions.ts` — Current Questions Agent JSON schema (zod) and Supabase insert pattern are reusable; extend to add `category_slugs` + `category_scores` fields.
- `pipeline/src/lib/claude.ts` — `extractJson`, `TokenAccumulator`, `checkBudget` — unchanged.

### Established Patterns
- Agent prompt returns JSON validated with Zod before any DB write — continue this pattern for new multi-category response shapes.
- Service-role key bypasses RLS for all pipeline writes — unchanged.
- Slug-based identity (categories table has `slug UNIQUE`) — already stable; agent returning slugs maps cleanly to existing lookups.

### Integration Points
- `question_plays` (existing) feeds `observed_score` — the nightly refresh job reads from this table.
- Phase 2.2 RPCs (`random_published_questions`, `random_general_knowledge_questions`, `counts_by_category_difficulty`) all need updating to query `question_categories` instead of `questions.category_id` / `questions.difficulty`.
- `categories` table `slug` column is the interface between agent string output and DB UUIDs.
