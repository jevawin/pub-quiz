Build the Questions Agent and Fact-Check Agent -- the "output" side of the sequential pipeline.

Purpose: The Questions Agent reads stored Wikipedia sources and generates high-quality multiple-choice questions. The Fact-Check Agent independently verifies each question's answer against source material, assigning a verification score. Together they produce verified quiz content ready for publication.

Output: Two working agent scripts with unit tests, ready to be called by the pipeline orchestrator.

**pipeline/src/agents/questions.ts**: Create the Questions Agent per D-08 and PIPE-06.

Export `runQuestionsAgent(config: PipelineConfig, tokenAccumulator: TokenAccumulator): Promise<AgentResult>`.

Use `log()` from `../lib/logger.ts` instead of raw `console.log`/`console.error` for all output.

Logic:
1. Create Supabase and Claude clients
2. Find categories with sources but needing questions:
   - Query categories that have at least 1 source
   - For each, count existing questions
   - Select categories with < 10 questions
   - Limit to a reasonable batch (process up to `config.questionsBatchSize` total questions across categories)
3. For each selected category:
   - Fetch source content from the sources table (up to 3 sources for the category)
   - **Dedup context cap:** Fetch up to 20 most recent `question_text` values for this category (ORDER BY created_at DESC LIMIT 20). If more than 20 exist, include a note in the prompt: "There are N existing questions for this category -- avoid overlapping topics." This prevents the dedup context from growing unbounded as the question library expands.
   - Call Claude with model `config.claudeModelGeneration` (Sonnet):
     - System prompt: "You are an expert pub quiz question writer. Generate multiple-choice trivia questions based ONLY on the provided reference material. Each question must have exactly 1 correct answer and 3 plausible but incorrect distractors. Write a 2-3 sentence explanation for why the correct answer is right. Assign difficulty: 'easy' (common knowledge), 'normal' (requires some knowledge), 'hard' (specialist knowledge). Do NOT generate questions similar to the ones already listed. IMPORTANT: Generate questions ONLY from facts that appear in the provided text. Do not use your own knowledge to create questions or answers."
     - User prompt: Include category name, source text content, list of existing questions to avoid (capped at 20), request N questions (e.g., 5 per category)
     - Request structured output matching `QuestionBatchSchema`

   **Design note on truncation and fact-checking:** The source content is truncated to `config.wikipediaMaxContentLength` (default 3000 chars) by the Knowledge Agent when storing it. The Questions Agent's system prompt explicitly instructs Claude to generate questions ONLY from the provided text. The Fact-Check Agent receives the SAME stored source text. This means all facts in generated questions should be verifiable from the stored content. If a question references info beyond the truncated text, it violates the system prompt instruction and should be caught by the fact-checker as unverifiable (score 0 or 1).

   - Parse response with `QuestionBatchSchema.parse()`
   - Track tokens: `trackUsage(response, tokenAccumulator, SONNET_INPUT, SONNET_OUTPUT)`
   - Check budget: `checkBudget(tokenAccumulator, config.budgetCapUsd)`
4. **Post-validation for each generated question, wrapped in try/catch for per-item error handling:**
   - Verify distractors array has exactly 3 items (Zod handles this, but double-check)
   - Verify no distractor equals the correct_answer (case-insensitive comparison)
   - If validation fails, skip the question, log with `log('warn', ...)`, and increment failed count
   - If insert throws, catch the error, log it, increment `failed`, and continue to next question. Do NOT throw to the orchestrator for per-item failures.
5. Insert valid questions into questions table:
   - `category_id`: the category's UUID
   - `source_id`: the first source's UUID used for this batch
   - `question_text`, `correct_answer`, `distractors` (as JSONB), `explanation`, `difficulty`
   - `verification_score`: 0 (unverified)
   - `status`: 'pending'
6. Return `{ processed: questionsInserted, failed: questionsFailed }`
7. Only throw to the orchestrator if ALL items fail or if it's a `BudgetExceededError`.

**pipeline/src/agents/fact-check.ts**: Create the Fact-Check Agent per D-08, PIPE-07, and D-14.

Export `runFactCheckAgent(config: PipelineConfig, tokenAccumulator: TokenAccumulator): Promise<AgentResult>`.

Use `log()` from `../lib/logger.ts` instead of raw `console.log`/`console.error` for all output.

The Fact-Check Agent independently verifies answers using source text (RAG-style) -- NOT LLM-on-LLM per PIPE-07. It receives the source text that the question was based on and checks whether the correct answer is supported by that text.

Logic:
1. Create Supabase and Claude clients
2. Fetch all pending, unverified questions:
   - Query questions where `status = 'pending'` AND `verification_score = 0`
   - Join with sources table via source_id to get the source content
   - Process all pending questions (Haiku is cheap, per research batch size recommendation)
3. Group questions by source_id for efficiency (send multiple questions about the same source in one call)
4. **For each group (or batch of questions with the same source), wrap in try/catch for per-item error handling:**
   - Call Claude with model `config.claudeModelVerification` (Haiku):
     - System prompt: "You are a fact-checker for a pub quiz app. For each question, verify whether the stated correct answer is actually correct based ONLY on the provided reference text. Do NOT use your own knowledge -- only what the reference text states. Score verification strength: 0 = cannot verify from text, 1 = weakly supported, 2 = clearly supported, 3 = explicitly stated in text. If the answer contradicts the reference text, mark is_correct as false."
     - User prompt: Include the source text, then list each question with its correct_answer and distractors. Request structured output matching `FactCheckBatchSchema`.
   - Parse response with `FactCheckBatchSchema.parse()`
   - Track tokens: `trackUsage(response, tokenAccumulator, HAIKU_INPUT, HAIKU_OUTPUT)`
   - Check budget: `checkBudget(tokenAccumulator, config.budgetCapUsd)`
5. **For each fact-check result, wrap in try/catch for per-item error handling:**
   - If `is_correct === true`:
     - Update question: `verification_score = result.verification_score`, `status = 'verified'`
     - **Auto-publish threshold is score >= 3 (high confidence only).** If `result.verification_score >= 3`: also set `status = 'published'`, `published_at = new Date().toISOString()`
     - Questions with score 1-2 remain as `status = 'verified'` -- they are not auto-published. They await a future human review mechanism. This is a deliberate safety measure: a single Haiku call deciding to publish is only trusted at the highest confidence level.
   - If `is_correct === false`:
     - Update question: `verification_score = 0`, `status = 'rejected'`
   - If the update throws, catch the error, log it with `log('error', ...)`, increment `failed`, and continue to next result. Do NOT throw to the orchestrator for per-item failures.
6. Return `{ processed: verifiedCount, failed: rejectedCount }`
7. Only throw to the orchestrator if ALL items fail or if it's a `BudgetExceededError`.
