# Phase 02.1: Question Pipeline -- QA Agent & Source Relevance - Research

**Researched:** 2026-04-05
**Domain:** AI agent pipeline (Claude Haiku), question quality assurance, Wikipedia source filtering
**Confidence:** HIGH

## Summary

This phase adds a QA Agent as the 5th pipeline step and improves the Knowledge Agent's source relevance. The codebase is well-structured with clear patterns -- every agent follows the same `run*Agent()` function signature, uses `AgentResult { processed, failed }`, and handles errors per-item. The QA Agent slots cleanly into this pattern.

The main technical work is: (1) a new `qa.ts` agent that queries verified questions, sends them to Haiku for quality checks, rewrites fixable issues, and handles auto-publish logic previously in fact-check.ts; (2) modifications to `knowledge.ts` to build richer search queries from category hierarchy and filter sources through Haiku relevance scoring; (3) removing auto-publish from `fact-check.ts`; (4) adding QA as the 5th step in `run-pipeline.ts`.

**Primary recommendation:** Follow the existing agent pattern exactly. The QA Agent is structurally identical to the Fact-Check Agent -- fetch questions, batch by source, call Haiku, parse response, update database. The Knowledge Agent changes are surgical: add parent category lookup for search terms, add a Haiku call to filter fetched articles before inserting.

## Architecture Patterns

### Existing Agent Pattern (MUST follow)

Every agent in this pipeline follows the same structure. The QA Agent and Knowledge Agent changes must match this exactly.

```typescript
// Source: pipeline/src/agents/fact-check.ts (established pattern)
export interface AgentResult {
  processed: number;
  failed: number;
}

export async function runQaAgent(
  config: PipelineConfig,
  tokenAccumulator: TokenAccumulator,
): Promise<AgentResult> {
  const supabase = createSupabaseClient(config.supabaseUrl, config.supabaseServiceRoleKey);
  const claude = createClaudeClient(config.anthropicApiKey);
  
  let processed = 0;
  let failed = 0;
  let errors = 0;
  
  // 1. Fetch items to process
  // 2. Loop with per-item try/catch
  // 3. Call Claude, parse with Zod schema
  // 4. Update database
  // 5. Throw only if ALL items failed due to errors
  
  if (processed === 0 && errors > 0) {
    throw new Error(`All ${errors} QA checks failed due to errors`);
  }
  return { processed, failed };
}
```

**Key conventions from existing code:**
- `AgentResult` is defined in each agent file (not shared -- duplicated by convention)
- Supabase type assertions use `as never` and `as unknown as Promise<...>` workarounds
- `extractJson()` handles both fenced and raw JSON from Claude responses
- `BudgetExceededError` is re-thrown (not caught) to stop the pipeline
- Logger uses structured key-value objects: `log('info', 'message', { key: value })`
- Claude calls use `config.claudeModelVerification` for Haiku (QA Agent should use this)

### QA Agent Data Flow

```
Supabase: questions WHERE status IN ('verified') AND verification_score >= 1
    |
    v
Group by source_id (same pattern as fact-check.ts)
    |
    v
For each group: fetch source content, build prompt with questions
    |
    v
Haiku: score 4 quality dimensions, identify fixable vs broken
    |
    v
Fixable: rewrite question_text/distractors/explanation, update DB, keep verification_score
Broken: update status to 'rejected'
Passed (score >= 3): update status to 'published', set published_at
Passed (score 1-2): leave as 'verified' (no change needed)
```

### Knowledge Agent Enhancement Flow

```
Current:  category.name -> searchArticles() -> getArticleText() -> insert source
Enhanced: category.name + parent context -> searchArticles() -> getArticleText()
              -> Haiku relevance filter -> insert only relevant sources
              -> if < 2 results: retry with broader terms (one retry only)
```

### QA Agent Zod Schemas

The QA Agent needs response schemas following the `FactCheckBatchSchema` pattern:

```typescript
// New schemas for lib/schemas.ts
const QaResultSchema = z.object({
  question_id: z.string().uuid(),
  passed: z.boolean(),
  action: z.enum(['pass', 'rewrite', 'reject']),
  // Scores for each quality dimension (0-10 or similar)
  natural_language_score: z.number(),
  category_fit_score: z.number(),
  difficulty_calibration_score: z.number(),
  distractor_quality_score: z.number(),
  // Rewrite fields (optional, present when action = 'rewrite')
  rewritten_question_text: z.string().optional(),
  rewritten_distractors: z.array(z.string()).optional(),
  rewritten_explanation: z.string().optional(),
  reasoning: z.string(),
});

const QaBatchSchema = z.object({
  results: z.array(QaResultSchema),
});
```

### Single Call vs Two Calls (Claude's Discretion)

**Recommendation: single call.** The Fact-Check Agent already does scoring + action in one call. QA should do the same -- score all 4 dimensions, decide action (pass/rewrite/reject), and provide rewrites in one response. Two calls doubles cost and adds complexity for no clear benefit.

Haiku is cheap (~$0.001-0.003 per question batch) and capable enough to handle scoring + rewriting in one shot. The prompt should instruct it to only provide rewrite fields when action is 'rewrite'.

### Haiku Relevance Threshold (Claude's Discretion)

**Recommendation: 0.6.** This balances filtering out genuinely irrelevant articles while keeping borderline-useful ones. The threshold should be configurable via `RELEVANCE_THRESHOLD` env var so it can be tuned without code changes.

### QA Batch Size (Claude's Discretion)

**Recommendation: no separate batch size needed.** The QA Agent processes whatever Fact-Check produced in this run. There is no independent batch sizing -- it processes all verified questions (score >= 1). If the volume grows large, batch by source group (which the code naturally does when grouping by source_id).

### QA Rewrite Tracking (Claude's Discretion)

**Recommendation: add a `qa_rewritten` boolean column to the questions table.** This is lightweight, queryable, and lets you track rewrite rates without a separate log table. A migration adds the column with a default of `false`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON extraction from Claude | Custom regex | `extractJson()` from `lib/claude.ts` | Already handles fenced and raw JSON |
| Response validation | Manual field checks | Zod schemas in `lib/schemas.ts` | Type-safe, descriptive errors, established pattern |
| Token tracking | Manual arithmetic | `trackUsage()` + `TokenAccumulator` | Already handles cost rates and accumulation |
| Budget enforcement | If-statement checks | `checkBudget()` + `BudgetExceededError` | Integrates with pipeline-level error handling |
| Category hierarchy queries | Recursive CTE | Single query with JOIN on `parent_id` | Categories are max depth 3, single self-join is enough for parent context |

## Common Pitfalls

### Pitfall 1: Fact-Check Test Breakage
**What goes wrong:** Existing fact-check tests assert auto-publish behaviour (lines 242-251 of fact-check.test.ts). When you remove auto-publish from fact-check.ts, these tests will fail.
**Why it happens:** D-03 moves auto-publish to QA Agent. The fact-check agent should now only set `status: 'verified'` (never 'published').
**How to avoid:** Update fact-check tests at the same time as the code change. The test on line 242 ("auto-publishes questions with score >= 3") must change to assert `status: 'verified'` instead of `status: 'published'`.
**Warning signs:** `vitest run` fails on fact-check tests after modifying the agent.

### Pitfall 2: Question Status Transitions
**What goes wrong:** QA Agent tries to publish a question that Fact-Check already set to 'verified', but Supabase update silently fails because of a status check constraint.
**Why it happens:** The questions table has a CHECK constraint: `status IN ('pending', 'verified', 'rejected', 'published')`. There is no transition constraint -- any status can move to any other. But you need to query the right starting status.
**How to avoid:** QA Agent should query `WHERE status = 'verified'` (not 'pending'). Fact-Check sets verified questions to 'verified'. QA then moves eligible ones to 'published'.

### Pitfall 3: Knowledge Agent TokenAccumulator Not Passed
**What goes wrong:** Knowledge Agent currently doesn't receive a `TokenAccumulator` (see line 75 of run-pipeline.ts: `runKnowledgeAgent(config)` -- no accumulator). But now it needs one for Haiku relevance filtering.
**Why it happens:** Phase 1 didn't need Claude calls in the Knowledge Agent.
**How to avoid:** Update `runKnowledgeAgent` signature to accept `TokenAccumulator` and pass it from `run-pipeline.ts`. This is a breaking change to the function signature.

### Pitfall 4: Missing Pipeline Stats Columns
**What goes wrong:** QA Agent results can't be recorded in `pipeline_runs` because the table has no QA-specific columns.
**Why it happens:** The table was designed for 4 agents, not 5.
**How to avoid:** Add a migration (00002) with columns like `questions_qa_passed`, `questions_qa_rewritten`, `questions_qa_rejected`. Update `database.types.ts` manually to match. Update the success record in `run-pipeline.ts`.

### Pitfall 5: Rewriting Distractors Breaks the Count Constraint
**What goes wrong:** QA Agent rewrites distractors but sends 2 or 4 instead of exactly 3. Database rejects with `chk_distractors_count` violation.
**Why it happens:** Haiku might not strictly follow the "exactly 3 distractors" instruction.
**How to avoid:** Validate rewritten distractors length in code before updating. The Zod schema should enforce `.length(3)` on the rewritten distractors array. If validation fails, skip the rewrite and pass the question as-is.

### Pitfall 6: Category Hierarchy Query Returns No Parent
**What goes wrong:** Knowledge Agent tries to build a richer query by fetching the parent category, but top-level categories have `parent_id = null`.
**Why it happens:** Categories at depth 0 have no parent.
**How to avoid:** Handle null parent gracefully -- fall back to just the category name for top-level categories. Also fetch child categories for context when parent is null.

### Knowledge Agent: Category Hierarchy Query

```typescript
// Fetch category with parent context for richer search queries (D-07)
const { data: categoryWithParent } = await supabase
  .from('categories')
  .select('id, name, slug, parent_id')
  .eq('id', category.id)
  .single();

let searchTerms = category.name;

if (categoryWithParent?.parent_id) {
  const { data: parent } = await supabase
    .from('categories')
    .select('name')
    .eq('id', categoryWithParent.parent_id)
    .single();
  
  if (parent) {
    searchTerms = `${category.name} ${parent.name}`;
  }
}

// Also fetch children for additional context
const { data: children } = await supabase
  .from('categories')
  .select('name')
  .eq('parent_id', category.id)
  .limit(3);

if (children && children.length > 0) {
  searchTerms += ' ' + children.map(c => c.name).join(' ');
}
```

### Knowledge Agent: Haiku Relevance Filter

```typescript
// Source relevance scoring with Haiku (D-06)
const relevancePrompt = `You are evaluating whether a Wikipedia article is relevant to a specific quiz category.

Category: ${category.name}
Article title: ${title}
Article excerpt: ${content.slice(0, 500)}

Rate the relevance of this article to the category on a scale of 0.0 to 1.0.
- 1.0 = directly about this category topic
- 0.5 = somewhat related
- 0.0 = completely unrelated

Return JSON: {"relevance": <number>, "reasoning": "<brief explanation>"}`;

const response = await claude.messages.create({
  model: config.claudeModelVerification,  // Haiku
  max_tokens: 256,
  messages: [{ role: 'user', content: relevancePrompt }],
});

trackUsage(response, tokenAccumulator, HAIKU_INPUT, HAIKU_OUTPUT);
checkBudget(tokenAccumulator, config.budgetCapUsd);
```

### Knowledge Agent: Fallback Search (D-08)

```typescript
// Fallback when initial search returns < 2 results
const titles = await searchArticles(searchTerms, config.wikipediaUserAgent, 5);

if (titles.length < 2) {
  // Drop most specific term, use parent category or broader terms
  const fallbackTerms = parentName ?? category.name.split(' ')[0];
  const fallbackTitles = await searchArticles(fallbackTerms, config.wikipediaUserAgent, 5);
  titles.push(...fallbackTitles);
}
```

    .from('questions')

}

## QA Agent Extended AgentResult

The standard `AgentResult { processed, failed }` doesn't capture rewrites. Options:

**Recommendation:** Extend QA Agent's return type to include `rewritten` count for observability, but keep the `AgentResult` interface contract for pipeline compatibility.

```typescript
interface QaAgentResult extends AgentResult {
  rewritten: number;
}
```

## Open Questions

1. **Question status after QA rejection**
   - What we know: D-02 says QA rejects "truly broken" questions. The status should go to 'rejected'.
   - What's unclear: Should QA-rejected questions have a different indicator from Fact-Check-rejected ones? Currently `status: 'rejected'` is the only option.
   - Recommendation: Use the same 'rejected' status. The `qa_rewritten` column distinguishes questions that went through QA. For rejection tracking, the pipeline_runs stats are sufficient.

2. **QA Agent prompt complexity**
   - What we know: 4 quality dimensions need scoring, plus optional rewrites.
   - What's unclear: Whether Haiku handles all this reliably in a single prompt.
   - Recommendation: Start with single prompt. If quality is poor, split into score-then-rewrite as a follow-up.
