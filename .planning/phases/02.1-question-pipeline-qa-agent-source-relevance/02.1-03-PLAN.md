**System prompt for QA Agent:**
```
You are a quality assurance reviewer for a pub quiz app. For each question, score these 4 dimensions (0-10):
1. Natural Language Quality: Is the question clearly written? Good grammar? Appropriate length (not too short, not too long)? No awkward phrasing?
2. Category Fit: Does this question belong in the stated category? Is it relevant?
3. Difficulty Calibration: Does the marked difficulty (easy/normal/hard) match how hard this question actually is?
4. Distractor Quality: Are the 3 wrong answers plausible but clearly wrong? No trick answers? No obviously absurd options?

For each question, decide an action:
- "pass": Question is good as-is (all scores >= 5)
- "rewrite": Question has fixable issues. Provide rewritten text. Only rewrite phrasing/distractors/explanation -- do NOT change the factual content or correct answer.
- "reject": Question is fundamentally broken (wrong category, nonsensical, unanswerable). Cannot be fixed by rewriting.

IMPORTANT: When rewriting distractors, you MUST provide exactly 3 distractors.
```

**QA Agent data flow:**
1. Fetch questions: `WHERE status = 'verified' AND verification_score >= 1`
2. Group by source_id (same pattern as fact-check)
3. For each source group: fetch source content, build prompt with questions + source text
4. Call Claude Haiku, parse with QaBatchSchema
5. For each result:
   - `action === 'pass'`:
     - If `verification_score >= 3`: update `status: 'published'`, `published_at: new Date().toISOString()` (per D-03)
     - If `verification_score < 3`: no DB update needed (stays verified, per D-04)
     - Increment processed
   - `action === 'rewrite'`:
     - Validate rewritten_distractors has exactly 3 items (if provided). If not, skip the rewrite and treat as 'pass'.
     - Update DB: `question_text`, `distractors`, `explanation` (if rewritten versions provided), set `qa_rewritten: true`
     - Keep existing `verification_score` (D-05) -- do NOT change it
     - Then apply the same publish logic as 'pass' (score >= 3 => publish, score < 3 => stay verified)
     - Increment processed AND rewritten
   - `action === 'reject'`:
     - Update `status: 'rejected'`
     - Increment failed
6. Return `{ processed, failed, rewritten }`
7. If all items had errors (processed === 0 && errors > 0), throw

Use the same Supabase type assertion pattern as fact-check.ts: `as never` for update data, `as unknown as Promise<...>` for the update result.
